## 오늘 공부한 것

- 포인터와 다차원 배열, 문자열
- 프로그래머스 문제 풀기

## 핵심 개념

**배열의 "붕괴(decay)"**
- 배열 이름은 원래 "여러 칸이 모인 덩어리"(예: `int[5]` 타입)이지, 포인터가 아님
- 하지만 `sizeof`, `&` 뒤가 아닌 거의 모든 상황(대입, 산술연산, 역참조 대상, 함수 인자 등)에서 값으로 쓰이면, 자동으로 "그 배열의 첫 번째 원소를 가리키는 포인터"로 변환됨 → 이 현상을 붕괴라고 함
- `+`, `*` 같은 연산자는 "배열 덩어리 전체"를 직접 다룰 수 없어서, 연산이 필요한 순간마다 배열을 "시작 주소(포인터)"로 즉석 변환해서 산술 계산(오프셋 이동)으로 처리하는 것
- 이건 어딘가 저장된 "연결(포인터)"을 따라가는 게 아니라, 계산이 필요한 순간마다 "시작 주소 + 오프셋"을 그때그때 계산하는 것
- 붕괴 안 되는 예외: `sizeof(arr)`(배열 전체 크기 계산), `&arr`(배열 전체를 가리키는 포인터), 문자열 리터럴 초기화의 특수 케이스

**붕괴가 일어나는 지점 — `arr2[i][j]` 분해 예시**
```c
arr2[i][j]  =  *(*(arr2 + i) + j)

① arr2 + i     ← "+" 연산 위해 arr2가 붕괴 (int[3][4] → int(*)[4])
② *(arr2+i)    ← 역참조 결과: arr2[i] 자체 (아직 배열, int[4])
③ (그 결과)+j   ← "+" 연산 위해 arr2[i]가 다시 붕괴 (int[4] → int*)
④ *(...)       ← 역참조로 진짜 int 값 도출
```
- 차원이 늘어날수록 붕괴가 그만큼 더 반복됨 (3차원이면 3번, N차원이면 N번)

**주소·값 표현 정리**

1차원 배열 (`int arr[5] = {1,2,3,4,5};`)

| 구하려는 것 | 표현식 | 타입 |
|---|---|---|
| `arr[0]`의 값 | `*arr`, `arr[0]` | `int` |
| `arr[0]`의 주소 | `arr`, `&arr[0]` | `int*` |
| 배열 전체의 주소 | `&arr` | `int(*)[5]` |

2차원 배열 (`int arr[3][4];`)

| 구하려는 것 | 표현식 | 타입 |
|---|---|---|
| `arr[0]`의 값 (0행 전체) | `*arr`, `arr[0]` | `int[4]` (배열!) |
| `arr[0]`의 주소 | `arr`, `&arr[0]` | `int(*)[4]` |
| `arr` 배열 전체의 주소 | `&arr` | `int(*)[3][4]` |
| `arr[0][0]`의 값 | `arr[0][0]`, `*arr[0]`, `**arr` | `int` |
| `arr[0][0]`의 주소 | `&arr[0][0]`, `arr[0]`, `*arr` | `int*` |
| `arr[i][j]`의 값 | `arr[i][j]`, `*(arr[i]+j)`, `*(*(arr+i)+j)` | `int` |
| `arr[i][j]`의 주소 | `&arr[i][j]`, `arr[i]+j` | `int*` |

3차원 배열 (`int arr3[2][3][4];`)

| 단계 | 표현 | 타입 |
|---|---|---|
| `arr3` | 붕괴 → `int(*)[3][4]` | i번째 "면" 가리킴 |
| `arr3[i]` | 붕괴 → `int(*)[4]` | i면 j행 가리킴 |
| `arr3[i][j]` | 붕괴 → `int*` | 원소 가리킴 |
| `arr3[i][j][k]` | `int` | 진짜 값 |

**치환법 — 다차원 배열 수식 빠르게 계산하기**
- `arr[i]`가 붕괴되면 1차원 배열과 똑같은 `int*` 타입이 되는 걸 이용
- 1단계: `arr[i]`를 `a`로 치환 (또는 동치 표현인 `*(arr+i)`로 치환해도 무방)
- 2단계: 1차원 공식 그대로 적용 → 값: `a[j]` 또는 `*(a+j)` / 주소: `a+j` 또는 `&a[j]`
- 3단계: `a`를 다시 `arr[i]`로 되돌림 → 값: `arr[i][j]` 또는 `*(arr[i]+j)` / 주소: `arr[i]+j` 또는 `&arr[i][j]`
- 동치 표현끼리는 자유롭게 교환 가능 (`arr[i]` ≡ `*(arr+i)`이므로 어느 쪽을 대입해도 결과 동일)

**선언 ≠ 연결(대입)**
```c
int (*p)[4];   // ① 선언: "4칸배열 주소를 담을 상자" 규격만 정함 (빈 상자)
p = arr2;      // ② 대입: 이제 실제 주소가 들어감 (진짜 연결은 여기서 발생)
```

**함수 매개변수에서의 동치 관계**
```c
void func(int (*parr)[4]) { ... }
void func(int parr[][4])  { ... }   // 완전히 동일
void func(int parr[3][4]) { ... }   // 3은 무시됨, 역시 동일
```
- 단, 이 동치는 함수 매개변수 자리에서만 성립. 일반 변수 선언에서 `int parr[][4]`는 포인터가 아니라 진짜 배열을 뜻함(초기화 필수)

**함수로 문자열 배열 데이터 넘기기**
- `str`을 그냥 넘기면 `char*`로 붕괴됨 → 함수는 `char *p` 또는 `char p[101]`로 받으면 됨 (가장 흔히 쓰는 방식)
- `&str`(배열 전체의 주소)을 넘기는 경우에는 함수가 `char (*p)[101]`로 받아야 함
- 어느 쪽이든 `p`는 `str`(또는 `&str`)과 같은 주소값을 저장하고 있어서, 이름만 다를 뿐 실제로는 같은 메모리를 가리키는 별개의 변수

**문자열 리터럴 vs 문자열 배열**
- `char *p = "hello";` (리터럴, 수정 불가)
  - 가능: `printf("%s", p);`, `strlen(p);`, `strcmp(p, "abc");`, `p = "world";`(다른 문자열을 가리키도록 변경)
  - 불가능: `p[0] = 'H';`, `strcpy(p, "world");`, `strcat(p, "!!!");` → 문자열 리터럴의 내용을 수정하려 하기 때문
- `char str[20] = "hello";` (배열, 수정 가능)
  - 가능: `str[0] = 'H';`, `strcpy(str, "world");`, `strcat(str, "!!!");`, `strlen(str);`, `strcmp(str, "abc");`
  - 불가능: `str = "world";` → 배열 이름은 다른 주소를 가리키도록 변경할 수 없기 때문
- 문자열 함수별 사용 가능 여부

| 함수 | 리터럴 | 배열 |
|---|---|---|
| `strlen()` | ✅ | ✅ |
| `strcmp()` | ✅ | ✅ |
| `strchr()` | ✅ | ✅ |
| `strcpy()` (목적지) | ❌ | ✅ |
| `strcat()` (목적지) | ❌ | ✅ |

- `strcpy(dest, src)`, `strcat(dest, src)`에서 dest(첫 번째 인자)는 반드시 수정 가능한 배열이어야 하고, src(두 번째 인자)는 리터럴이든 배열이든 상관없음
